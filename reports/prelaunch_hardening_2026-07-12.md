# Pre-launch hardening — 2026-07-12

## 1. Demo rate limits reverted to safe public values (LAUNCH-BLOCKER, closed)

**Before**: `DEMO_PER_IP_HOUR_LIMIT=1000`, `DEMO_DAILY_REQUEST_CAP=5000` (env var
overrides on the Cloud Run service, set 2026-07-04 for testing per the code's own
comment: *"set it on the Cloud Run service to raise the limit for testing, then
remove/reset it before opening to the public"*).

**Action**: removed both env vars via
`gcloud run services update asa-stylist-api --remove-env-vars=DEMO_PER_IP_HOUR_LIMIT,DEMO_DAILY_REQUEST_CAP`
(surgical removal — confirmed the other 9 env vars/secrets on the service were
untouched). New revision: `asa-stylist-api-00070-5f9`.

**After (now live)**: falls back to the code's own defaults —
`DEMO_PER_IP_HOUR_LIMIT=10` (`api/demo/guards.py:22`), `DEMO_DAILY_REQUEST_CAP=200`
(`api/demo/guards.py:29`). `RATE_LIMIT_PER_MINUTE` (a separate, non-demo-mode rate
path, `api/rate_limit.py`) was never overridden and remains at its own default of 10/min.

**Live proof**: fired 15 consecutive `/chat` requests against the production URL.
Request #2 already returned `429 {"detail": "Rate limit exceeded"}` (I'd made 8+
calls against this same IP earlier this hour during this session's own live-proof
scripts, so the 10/hour cap was already partially consumed) — every subsequent
request also 429'd. This unambiguously confirms the new, much lower limit is live
and enforced (previously this same request pattern would have needed >1000 calls to
trip anything).

**Note on tuning**: 10/hour per IP is quite tight for someone genuinely trying the
demo (a single search + one refinement already uses 2 requests). This is the
developer's own pre-existing coded default, not a new number I invented — reverting
to it is the literal, safe interpretation of "revert." If it proves too restrictive
for real demo traffic, raising it to something like 30-50/hour is a product/cost
tradeoff decision, not a security one, and belongs to a deliberate follow-up
decision, not a silent unilateral change made while "reverting."

## 2. NPM vulnerabilities patched (LAUNCH-BLOCKER item, closed)

**Before**: `npm audit` — 10 vulnerabilities (1 low, 8 moderate, 1 high). The high:
`ws` 8.0.0-8.20.1, transitive dependency, uninitialized memory disclosure
(GHSA-58qx-3vcg-4xpx) + memory exhaustion DoS (GHSA-96hv-2xvq-fx4p).

**Action**: `npm audit fix` (non-breaking, no `--force`) — 15 packages added, 47
removed, 33 changed.

**After**: 0 critical, 0 high, 0 low, **2 moderate remaining** (both
`postcss < 8.5.10` XSS-via-unescaped-`</style>`, GHSA-qx2v-qp2m-jg93, reached only
through `node_modules/next`'s own pinned copy).

**Why the remaining 2 were NOT force-fixed**: `npm audit fix --force` proposes
downgrading `next` to `9.3.3`. Currently installed: `next@15.5.18` (confirmed via
`npm ls next`). That's a six-major-version downgrade to a release that predates the
App Router entirely — this application's routes are 100% App Router
(`app/`, layouts, `opengraph-image.tsx`, etc.). Forcing it would break the entire
frontend to fix a moderate-severity issue whose real exploitability here is low (this
app's CSS is Tailwind-generated at build time from developer-controlled source, not
constructed from untrusted user input at runtime — the advisory's actual attack
vector doesn't apply cleanly to this app's usage pattern). Verdict: not worth it;
tracked as a known, deliberate residual finding, not an oversight.

**Verification**: `npm run build` — passes cleanly, identical route set and bundle
sizes to before the fix. Committed (`6a48674`, `package-lock.json` only — no manual
edits, generated by the tool). Deployed to production
(`https://stylemaitri.vercel.app`, confirmed live via `curl` 200 on `/demo`).

## 3. Production observability (minimal, free-tier, closed)

**Starting point (better than my earlier MLOps audit gave credit for)**: the backend
already had proper structured JSON logging (`api/logging_config.py`) with a
`_JsonFormatter` surfacing `latency_ms`, `action`, `n_items`, `provider` as filterable
fields on every log line, and `api/routes/chat.py` already logs a `"chat turn
complete"` line with `n_items` per turn (both REST and WS paths), plus a generic
`access_log` middleware logging every request's method/path/status/latency. This data
was already flowing into Cloud Logging (Cloud Run ships container stdout there by
default, no setup) — it just wasn't turned into anything visible/alertable. **Zero
code changes were needed**; the gap was entirely GCP-side wiring.

**Added** (all via `gcloud logging`/`gcloud alpha monitoring`, all free-tier — Cloud
Logging's free tier is 50 GiB/month ingestion + log-based metrics, Cloud Monitoring's
free tier covers this traffic volume many times over):

1. `asa_chat_total_count` — counts every `/chat` request (any status). Denominator
   for the rates below.
2. `asa_chat_5xx_count` — counts `/chat` requests returning 5xx (server errors).
3. `asa_chat_ratelimit_count` — counts `/chat` requests returning 429 (per-IP hourly
   limit, daily request cap, or daily cost cap — the three demo guards from Item 1).
4. `asa_chat_empty_result_count` — counts completed chat turns where `n_items=0`
   (the "empty-result rate" the task asked for).
5. One alert policy, **"asa-stylist-api: chat 5xx errors"** — fires when
   `asa_chat_5xx_count > 0` in a 5-minute window. Created with **no notification
   channel** deliberately — I don't have authorization to guess an email/Slack target
   to send alerts to. It will show as an open incident in the Cloud Monitoring console
   immediately; adding a notification channel (email/SMS/Slack) is a 30-second UI
   action whenever you want it, and I didn't want to silently pick a destination on
   your behalf.

**Not added, and why (staying minimal per the instruction)**: request rate, HTTP
latency percentiles (p50/p95/p99), and instance/CPU/memory utilization are **already
free and automatic** on any Cloud Run service — no metric, log, or code needed. View
them at: Cloud Console → Cloud Run → `asa-stylist-api` → Metrics tab (or Cloud
Monitoring → Metrics Explorer → `run.googleapis.com/request_count` /
`request_latencies`). Re-implementing these would have been pure duplication.

**Live proof**: fired 3 fresh `/chat` requests (all 429'd, rate limit from Item 1
still active), then queried the Cloud Monitoring time-series API directly for all 4
new metrics.
- `asa_chat_ratelimit_count` and `asa_chat_total_count`: **confirmed capturing real
  data** — 3 events across two 1-minute buckets (2 + 1), exactly matching the 3 test
  calls.
- `asa_chat_5xx_count` and `asa_chat_empty_result_count`: showed zero series. This is
  the CORRECT state, not a wiring failure — no 5xx has actually occurred (all
  responses this session were 200 or 429), and a 429-rejected request never reaches
  the `"chat turn complete"` log line at all, so no empty-result event has fired
  either. Raw log query (`gcloud logging read`) independently confirmed the
  underlying `POST /chat 429` / `POST /chat 200` lines exist and are well-formed.

**Cost**: $0. All four metrics and the alert policy are GCP configuration objects, no
new compute, no new log volume beyond what already existed.
