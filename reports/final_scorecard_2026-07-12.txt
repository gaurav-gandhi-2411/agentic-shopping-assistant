SECTION B — TWO PRODUCTION FIXES THIS SESSION NOT VISIBLE IN SECTION A
(both live, both deployed, both proven via full-graph pytest + live-URL, NEITHER
detectable by eval_strict.py's harness by construction -- see Section A note)
====================================================================================

--- B1. Gender-heuristic-ordering fix (commit d71760b, deployed 00067-5sv) ---
BEFORE: any "<garment> for men" query where the garment itself (historically "kurta")
was listed as a gender-implying heuristic silently resolved to gender=women.
  parse_intent("printed kurta for men").gender  ->  "women"   (WRONG)
AFTER: explicit gender words always win over any garment-implied fallback.
  parse_intent("printed kurta for men").gender  ->  "men"     (correct)
Live proof (2026-07-11, post-deploy): "printed kurta for men" -> 4 of 5 results
explicitly men's kurtas (was 5 of 5 WOMEN's kurtas pre-fix, same day, same query).
8 new regression tests (tests/test_intent_parser.py). Side effect: eval_gate.py's
intent all-exact rose 92.4% -> 93.8% (one of the main model_eval_queries.yaml fixture
categories exercises this exact pattern).

--- B2. Occasion-term-pollution fix (commit bdf66a4, deployed 00068-sd2) ---
BEFORE: search_node's occasion-term retrieval-widening (RED 5b/D: appends garment
words like "lehenga sherwani kurta embellished festive" to broaden dense/BM25 recall
for garment-type-less occasion queries) fed that WIDENED string into the very next
block (facet auto-extraction), letting an injected word win a hard filter over what
the user actually typed.
  "lehenga for sangeet"           -> was hard-filtering to sherwani (men's garment)
  "wedding guest saree for women" -> was hard-filtering to sherwani too
  "haldi/mehendi outfit for women" -> was always hard-filtered to "lehenga" (the
                                       longest injected word), defeating RED 5b/D's
                                       own purpose of staying broad across types
AFTER: facet extraction scans raw_query only; the retrieval-widening itself is
untouched. Live proof (2026-07-12, post-deploy):
  "lehenga for sangeet"            -> 4 items, types=[lehenga]           (fixed)
  "wedding guest saree for women"  -> 3 items, types=[saree]             (fixed)
  "haldi outfit for women"         -> 2 items, types=[anarkali, dupatta] (fixed, was 1 type)
  "mehendi outfit for women"       -> 2 items, types=[dupatta, sharara]  (fixed, was 1 type)
3 new regression tests (tests/test_occasion_search_augmentation.py), run end-to-end
against the real unified index through the full agent graph.
